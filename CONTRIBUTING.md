## 📝 提交 Issue

### Issue 类型

- **Bug 报告**：发现错误
- **功能请求**：建议新功能
- **文档改进**：文档相关建议
- **问题咨询**：使用问题

### 提交规范

1. **搜索现有 Issue**
   - 在提交前搜索是否已有相关 Issue

2. **选择模板**
   - 使用提供的 Issue 模板

3. **清晰描述**
   - 标题：简洁明了
   - 详细描述问题
   - 提供复现步骤（如果是 Bug）
   - 提供相关环境信息

4. **添加标签**
   - 选择合适的标签

### Issue 模板

```markdown
## 问题描述
<!-- 清晰描述问题 -->

## 复现步骤
<!-- 详细说明如何复现问题 -->
1. 
2. 
3. 

## 期望结果
<!-- 期望发生什么 -->

## 实际结果
<!-- 实际发生了什么 -->

## 环境信息
- 操作系统：
- 浏览器/编辑器：
- 版本号：

## 截图
<!-- 如有，请添加截图 -->

## 其他信息
<!-- 其他相关信息 -->
```

## 🚀 提交 Pull Request

### PR 流程

1. **Fork 项目**
   ```bash
   git clone https://github.com/1803560007/markdown-writing-guide.git
   cd markdown-writing-guide
   ```

2. **创建分支**
   ```bash
   git checkout -b feature/your-feature-name
   # 或
   git checkout -b fix/your-fix-name
   ```

3. **修改代码**
   - 遵循代码规范
   - 添加必要的注释
   - 更新相关文档

4. **提交代码**
   ```bash
   git add .
   git commit -m "feat: 添加新功能"
   git push origin feature/your-feature-name
   ```

5. **创建 Pull Request**
   - 访问 GitHub 仓库
   - 点击 "New pull request"
   - 填写 PR 描述
   - 等待审查

### PR 模板

```markdown
## 描述
<!-- 描述你的改动 -->

## 类型
<!-- 选择一个类型 -->
- [ ] Bug 修复
- [ ] 新功能
- [ ] 文档更新
- [ ] 代码优化
- [ ] 其他

## 相关 Issue
<!-- 关联的 Issue 编号 -->
Closes #123

## 变更说明
<!-- 详细说明变更内容 -->

## 测试
<!-- 如何测试这些变更 -->
- [ ] 已测试
- [ ] 需要帮助测试

## 截图
<!-- 如有，请添加截图 -->

## 检查清单
- [ ] 代码遵循项目规范
- [ ] 文档已更新
- [ ] 测试通过
- [ ] 没有破坏性变更
```

## 📖 代码规范

### Markdown 规范

1. **标题层级**
   - 使用正确的标题层级（1-6 级）
   - 跳过层级是不允许的

2. **空行**
   - 标题前后各空一行
   - 段落之间空一行
   - 列表项之间空一行

3. **缩进**
   - 使用 2 个空格缩进
   - 不要使用 Tab

4. **链接**
   - 使用描述性链接文本
   - 避免使用"点击这里"

5. **图片**
   - 添加 alt 文本
   - 使用描述性文件名

6. **代码块**
   - 指定语言以获得语法高亮
   - 代码块前后空一行

7. **表格**
   - 对齐表头和内容
   - 保持格式一致

8. **引用**
   - 引用内容前后空一行
   - 嵌套引用使用正确的缩进

### 文档风格

1. **语言**
   - 使用简体中文
   - 保持术语一致
   - 避免口语化表达

2. **结构**
   - 逻辑清晰
   - 层次分明
   - 便于阅读

3. **示例**
   - 提供完整的示例代码
   - 示例要可运行
   - 添加必要的注释

4. **链接**
   - 内部链接使用相对路径
   - 外部链接使用完整 URL
   - 避免死链

## 📝 文档规范

### 文件命名

- 使用小写字母和连字符
- 示例：`getting-started.md`、`best-practices.md`

### 文件结构

```
markdown-writing-guide/
├── README.md              # 主文档
├── CONTRIBUTING.md        # 贡献指南
├── CHANGELOG.md          # 更新日志
├── docs/                 # 文档目录
│   ├── basics/          # 基础语法
│   ├── advanced/        # 进阶技巧
│   ├── examples/        # 示例
│   └── templates/       # 模板
└── images/              # 图片目录
```

### 文档头部

每个文档应包含以下信息：

```markdown
# 标题

**作者：** 作者名  
**更新时间：** YYYY-MM-DD  
**版本：** 1.0.0

<!-- 目录 -->
```

### 目录

长文档应包含目录：

```markdown
## 📚 目录

- [章节 1](#章节-1)
- [章节 2](#章节-2)
```

## 🔤 提交信息规范

### 格式

```
<类型>: <描述>

[可选的正文]

[可选的脚注]
```

### 类型

- **feat**: 新功能
- **fix**: Bug 修复
- **docs**: 文档更新
- **style**: 代码格式调整
- **refactor**: 代码重构
- **perf**: 性能优化
- **test**: 测试相关
- **chore**: 构建/工具相关
- **revert**: 回滚提交

### 示例

```
feat: 添加 Mermaid 图表示例

添加了流程图、时序图和甘特图的示例代码。
更新了相关文档说明。

Closes #45
```

```
fix: 修正表格对齐问题

修复了表格列对齐不正确的问题。
更新了最佳实践章节。

Related to #32
```

```
docs: 更新快速开始章节

添加了更多安装示例和常见问题解答。
优化了文档结构。
```

### 规范

- **英文小写**：类型使用小写英文
- **冒号后空格**：类型和描述之间有空格
- **描述简洁**：使用祈使句，不超过 50 字
- **正文详细**：解释为什么和怎么做（可选）
- **关联 Issue**：使用 `Closes #123` 或 `Related to #45`

## 🎨 代码风格

### Markdown 风格

1. **使用 CommonMark 标准**
   - 遵循 CommonMark 规范
   - 避免使用非标准语法

2. **保持一致性**
   - 统一的标题格式
   - 统一的列表格式
   - 统一的代码块格式

3. **可读性**
   - 每行不超过 80 字符
   - 适当换行
   - 使用空行分隔

### 示例代码风格

1. **完整性**
   - 提供完整的示例
   - 包含必要的导入
   - 包含使用说明

2. **注释**
   - 关键代码添加注释
   - 解释复杂逻辑
   - 标注注意事项

3. **测试**
   - 示例代码应可运行
   - 提供测试步骤
   - 说明预期结果

## 🧪 测试

### 文档测试

1. **链接检查**
   - 所有链接应可访问
   - 内部链接应正确跳转

2. **代码示例**
   - 示例代码应正确
   - 格式应正确

3. **拼写检查**
   - 无错别字
   - 语法正确

### 工具推荐

- **markdownlint**: Markdown 语法检查
- **link-checker**: 链接检查工具
- **spell-checker**: 拼写检查工具
